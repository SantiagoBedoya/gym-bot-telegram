## Project Overview

Gym-Bot is an AI-powered personal training assistant Telegram bot built in Go. It uses OpenAI's Responses API (agentic loop with tool use) to manage gym routines, track workout sessions, and provide progressive overload recommendations.

## Commands

```bash
# Run the bot
go run ./cmd/bot/main.go

# Build binary
go build -o gym-bot ./cmd/bot/

# Regenerate database query code (after editing SQL in internal/db/queries/)
sqlc generate

# Manage dependencies
go mod tidy
```

## Architecture

**Request flow:**
```
Telegram message → TelegramHandler (auth check) → OpenAIService.Chat()
    → [Agentic loop: OpenAI ↔ ToolDispatcher ↔ PostgreSQL]
    → Response sent to user, bot_state saved
```

**Key components:**

- `cmd/bot/main.go` — Startup: loads config → connects DB + runs migrations → wires services → starts Telegram polling
- `internal/config/config.go` — Loads required env vars: `BOT_TOKEN`, `OPENAI_API_KEY`, `DATABASE_URL`, `AUTHORIZED_USER_ID`, `OPENAI_MODEL`
- `internal/services/openai_service.go` — Implements the agentic loop using OpenAI Responses API. Persists `previous_response_id` via `bot_state` table to maintain conversation continuity across requests.
- `internal/tools/tools.go` — Defines the 5 tools available to the AI and dispatches calls to database operations.
- `internal/db/` — `pool.go` creates the pgx connection pool and auto-runs embedded SQL migrations. The `*.sql.go` files are **auto-generated by sqlc** — edit queries in `internal/db/queries/` and run `sqlc generate` instead.

**AI Tools (OpenAI function-calling):**
1. `save_routine` — Upsert a routine + exercises (cascade delete + reinsert)
2. `get_routine` — Fetch all exercises in a routine
3. `get_exercise_history` — Recent set-by-set performance data for progressive overload
4. `log_session_sets` — Record a completed workout session (wrapped in a transaction)
5. `get_last_session_summary` — Retrieve metrics from the most recent session

**Progressive overload logic** is handled entirely by the AI system prompt (double progression model): increase reps → increase weight → deload on high RPE.

## Database

PostgreSQL via Neon (serverless). Schema is managed through embedded migrations in `internal/db/migrations/`. Migrations run automatically on startup and are idempotent.

Tables: `routines`, `exercises`, `sessions`, `session_sets`, `bot_state`

## Important Notes

- Do **not** edit `internal/db/*.sql.go` files — they are generated by sqlc from `internal/db/queries/*.sql`.
- Only one Telegram user (set by `AUTHORIZED_USER_ID`) can interact with the bot.
- The bot maintains conversation continuity using OpenAI's `previous_response_id`, stored in the `bot_state` table under the key `last_response_id`.
